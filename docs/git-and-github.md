# Committing to Git and pushing to GitHub

This walks a developer from a fresh copy of this repository to a first commit and
a pushed remote. It assumes [Git](https://git-scm.com/) is installed. The
optional `gh` steps use the [GitHub CLI](https://cli.github.com/).

## 0. One-time chores before publishing

- Open `LICENSE` and replace `<COPYRIGHT HOLDER>` with your name or organization.
- Confirm `git --version` works and your identity is set:

  ```bash
  git config --global user.name  "Your Name"
  git config --global user.email "you@example.com"
  ```

## 1. What is and isn't tracked

`.gitignore` already excludes the things that should never be committed:

- `node_modules/` (regenerated by `npm install`)
- `.venv/`, `__pycache__/`, `.pytest_cache/`, `.ruff_cache/`
- `.env` and anything holding secrets (your `DATABASE_URL` lives in the
  environment, never in the repo)
- build artifacts and `*.zip`

`package-lock.json` **should** be committed — it pins exact dependency versions.

## 2. Initialize and make the first commit

From the repository root:

```bash
git init
git add .
git commit -m "Initial commit: warehouse layout editor v0.1.0"
```

Tag the release so the version in `VERSION` is reachable in history:

```bash
git tag -a v0.1.0 -m "v0.1.0"
```

## 3. Create the remote and push

### Option A — GitHub CLI (one command)

```bash
gh repo create warehouse-layout-editor --private --source=. --remote=origin --push
git push --tags
```

### Option B — Manual

1. Create an empty repository on github.com (no README/license — this repo has
   them). Copy its URL.
2. Wire it up and push:

   ```bash
   git branch -M main
   git remote add origin git@github.com:<your-username>/warehouse-layout-editor.git
   git push -u origin main
   git push --tags
   ```

   Use the `https://github.com/<your-username>/warehouse-layout-editor.git` form
   if you authenticate with HTTPS instead of SSH.

## 4. Continuous integration

`.github/workflows/ci.yml` runs automatically on every push and pull request to
`main`. It lints and tests both the Python (`ruff`, `pytest`) and the web
(`prettier`, `eslint`, `node --test`) sides. Watch the **Actions** tab after your
first push; the badge should go green.

## 5. Everyday workflow

```bash
git checkout -b feature/short-description
# ...make changes...
ruff check . && ruff format --check . && pytest      # python gate
node --test                                          # js gate
git add -p
git commit -m "Describe the change"
git push -u origin feature/short-description
```

Open a pull request on GitHub; merge once CI is green.

## Releasing a new version

1. Bump `VERSION` and the `version` fields in `pyproject.toml` and `package.json`.
2. Add a dated section to `CHANGELOG.md`.
3. Commit, then tag: `git tag -a vX.Y.Z -m "vX.Y.Z" && git push --tags`.

Remember the project version (semver) is separate from the layout
`schemaVersion`, which versions the data format.
